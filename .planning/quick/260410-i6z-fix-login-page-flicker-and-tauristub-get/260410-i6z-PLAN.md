Root cause analysis:

**Bug 1 — Login page refresh loop:**
`apiRequest` in `client.ts` line 21 does `window.location.href = '/login'` on any 401 response.
This is a hard browser navigation (full page reload). The AuthProvider (wrapping all routes including
/login) calls `getCurrentUser()` on mount, which calls `apiRequest('/me')`. When unauthenticated,
the 401 triggers `window.location.href = '/login'` — but we are ALREADY on /login. This creates:
  /login -> mount -> getCurrentUser -> 401 -> window.location.href='/login' -> full reload -> repeat

The fix: Remove the `window.location.href` redirect from apiRequest. The auth flow is already
handled correctly by AuthProvider + ProtectedRoute: AuthProvider catches the 401 error, sets
user=null, and ProtectedRoute renders `<Navigate to="/login">`. The hard redirect in apiRequest
is redundant AND harmful.

**Bug 2 — getSettings console warning:**
ThemeProvider (lines 189-202) calls `getSettings()` from the tauri stub on mount to "sync with
backend settings". The stub just logs a warning and returns null. The `setTheme` callback (line 209)
also calls `getSettings()` to read-then-update. Since there is no server settings API, these calls
should be removed — localStorage is already the source of truth for theme settings.
</context>
