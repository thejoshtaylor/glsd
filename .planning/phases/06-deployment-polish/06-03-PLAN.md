1. **Add the `write_env` function** before `main()`:

```sh
write_env() {
    ENV_DIR="${GSD_INSTALL_DIR:-$HOME/.gsd-cloud}"
    # If GSD_INSTALL_DIR ends with /bin, go up one level for .env
    case "$ENV_DIR" in
        */bin) ENV_DIR=$(dirname "$ENV_DIR") ;;
    esac
    ENV_FILE="$ENV_DIR/.env"

    if [ -f "$ENV_FILE" ]; then
        say "  .env already exists at $ENV_FILE, skipping"
        return
    fi

    # Get server URL from env or prompt
    if [ -z "${GSD_SERVER_URL:-}" ]; then
        if [ -t 0 ]; then
            printf '%s' "GSD Server URL (e.g. https://gsd.example.com): "
            read -r GSD_SERVER_URL
            if [ -z "$GSD_SERVER_URL" ]; then
                err "GSD_SERVER_URL is required"
            fi
        else
            err "GSD_SERVER_URL not set and stdin is not a terminal. Set GSD_SERVER_URL before running."
        fi
    fi

    # Basic URL validation -- must start with http:// or https://
    case "$GSD_SERVER_URL" in
        http://*|https://*)
            : # valid
            ;;
        *)
            err "GSD_SERVER_URL must start with http:// or https:// (got: $GSD_SERVER_URL)"
            ;;
    esac

    mkdir -p "$ENV_DIR"
    cat > "$ENV_FILE" <<ENVEOF
# GSD Cloud daemon configuration
# Edit this file to change settings. No need to re-run the installer.
GSD_SERVER_URL=$GSD_SERVER_URL
ENVEOF
    say "  wrote $ENV_FILE"
}
```

2. **Call `write_env` in `main()`** -- insert the call after the `ensure_path_hint` call and before the final "Next steps" section. The flow becomes:

Note: The `ENV_DIR` variable is set inside `write_env`. To make it available in the "Next steps" block, either: (a) compute `ENV_DIR` earlier in `main()`, or (b) set it as a global. Simplest approach: compute `ENV_DIR` at the start of `main()` and use it in both places.


Then `write_env` can reference $ENV_DIR directly (remove the local computation from write_env and use the one from main).

**Security per RESEARCH threat model:** The URL is validated to start with `http://` or `https://` before writing to .env. The value is never interpolated into shell commands -- only written to a flat config file. The `-t 0` check prevents the script from hanging when piped.

## Trust Boundaries

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-10 | Tampering | Script injection via GSD_SERVER_URL | mitigate | URL validated with case pattern to start with `http://` or `https://`. Value written to .env via heredoc (not shell interpolation into commands). No `eval` or command substitution on the value. |
| T-06-11 | Tampering | MITM during install script download | accept | Existing threat -- install.sh is downloaded via `curl -fsSL https://...`. HTTPS provides transport security. Checksum verification of the binary is already implemented. The install script itself is not checksummed (standard for curl-pipe-sh installers). |
| T-06-12 | Information Disclosure | .env file permissions | mitigate | `cat > "$ENV_FILE"` creates file with user's default umask. On most systems this is 0644. The .env contains only a URL (not secrets). If GSD_NODE_TOKEN is added later, file permissions should be tightened to 0600. |
| T-06-13 | Tampering | Unsafe temp file in install | accept | Existing `mktemp -d` with trap cleanup already in place. No new temp files created by this change. |
</threat_model>
