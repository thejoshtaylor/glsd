## Summary

Phase 1 restructures four existing projects (daemon, protocol-go, deployable-saas-template, gsd-vibe) into a two-top-level-directory monorepo (`server/` and `node/`) with all build pipelines passing. No feature code is added or modified.

The four projects currently sit as sibling directories at the repo root. The target layout moves the deployable-saas-template's `backend/` into `server/backend/`, replaces its `frontend/` with gsd-vibe into `server/frontend/`, moves `daemon/` into `node/daemon/`, and moves `protocol-go/` into `node/protocol-go/`. A root `pnpm-workspace.yaml` links the JS packages, and a root `go.work` links the Go modules.

**Primary recommendation:** Move files first, then fix all build configurations (go.work, pnpm-workspace.yaml, Dockerfiles, import paths, tsconfig). Verify each build pipeline independently before considering the phase complete.

### Pattern 1: pnpm Workspaces for JS

**What:** A root `pnpm-workspace.yaml` declares which directories contain JS packages. A root `package.json` provides workspace-level scripts.

**Root pnpm-workspace.yaml:**
```yaml
packages:
  - "server/frontend"
```
[VERIFIED: pnpm workspace docs at https://pnpm.io/workspaces]

**Root package.json:**
```json
{
  "name": "gsd-cloud",
  "private": true,
  "packageManager": "pnpm@9.15.0",
  "engines": {
    "node": ">=22.0.0"
  },
  "scripts": {
    "build": "pnpm -r build",
    "dev": "pnpm --filter server-frontend dev",
    "lint": "pnpm -r lint",
    "test": "pnpm -r test"
  }
}
```

**Key behavior:** `pnpm install` at root installs all workspace packages. `pnpm -r build` runs build in all packages. `pnpm --filter <name> <cmd>` runs in a specific package. [VERIFIED: pnpm docs]

**gsd-vibe package.json name change:** The `name` field in `server/frontend/package.json` should change from `"vcca"` to `"server-frontend"` (or `"gsd-cloud-frontend"`) so `--filter` commands are readable. [ASSUMED]

### Pattern 2: Go Workspaces for Go Modules

**What:** A root `go.work` file tells the Go toolchain to resolve local module paths instead of fetching from the module proxy.

**Root go.work:**
```
go 1.25.0

use (
    ./node/daemon
    ./node/protocol-go
)
```
[VERIFIED: `go help work` output, Go workspace docs]

**Key behavior:** When `go.work` exists, `go build ./...` from the repo root resolves `github.com/gsd-build/protocol-go` to `./node/protocol-go` instead of fetching v0.1.0 from the proxy. The `go.mod` files remain unchanged -- they still declare the remote module paths. The workspace overlay is purely local. [VERIFIED: Go workspace documentation]

**Critical:** Run `go work sync` after creating go.work to generate `go.work.sum`. [VERIFIED: `go help work sync`]

### Pattern 3: uv Workspace for Python Backend

**What:** The deployable-saas-template has a root `pyproject.toml` with `[tool.uv.workspace] members = ["backend"]`. This must be preserved.

**After restructure:** The `server/pyproject.toml` (moved from `deployable-saas-template/pyproject.toml`) maintains:
```toml

```
[VERIFIED: existing deployable-saas-template/pyproject.toml]

The `uv.lock` file at `server/` level must also be moved. Alembic.ini paths reference `app/alembic` which is relative to the backend directory -- these remain unchanged. [VERIFIED: alembic.ini content]

### Anti-Patterns to Avoid

- **Moving files with `mv` instead of `git mv`:** Loses git history. Always use `git mv` for tracked files so git can detect renames. [ASSUMED -- standard git best practice]
- **Modifying go.mod with `replace` directives:** go.work is the correct mechanism for local module linking. `replace` directives in go.mod pollute the module for downstream consumers. [VERIFIED: Go workspace docs recommend workspaces over replace]
- **Creating a root node_modules:** The root `package.json` should NOT have dependencies. Only workspace packages have dependencies. pnpm hoists to the root `node_modules` automatically. [VERIFIED: pnpm workspace behavior]
- **Changing Python import paths:** The backend uses `from app.X import Y` everywhere. Since the `app/` directory stays inside `backend/`, these imports remain valid. Do NOT change them. [VERIFIED: grep of import statements]

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| JS workspace linking | Symlinks or path hacks | `pnpm-workspace.yaml` | pnpm handles hoisting, deduplication, lockfile management |
| Go local module resolution | `replace` directives in go.mod | `go.work` | Workspaces are the official mechanism; replace directives are for downstream consumers |
| Python workspace | Manual PYTHONPATH manipulation | uv workspace (`pyproject.toml`) | Already configured in the template |
| Monorepo build orchestration | Turborepo / Nx / custom scripts | pnpm `-r` and `--filter` flags | Two deployment targets, no shared build caching needs |

## Common Pitfalls

### Pitfall 1: Docker Build Context Paths Break

**What goes wrong:** Dockerfiles reference paths relative to the build context (e.g., `COPY ./backend/scripts /app/backend/scripts`). After restructuring, the build context changes.
**Why it happens:** The template's `docker-compose.yml` sets `context: .` (the template root), which after restructure becomes `server/`. Dockerfiles use relative paths from that context.
**How to avoid:** After moving files, update `docker-compose.yml` build context to `context: .` (relative to `server/`) and verify all COPY paths in Dockerfiles still resolve. The backend Dockerfile copies from `backend/` relative to context, which is fine if context is `server/`. The frontend Dockerfile copies from `frontend/` relative to context, also fine.
**Warning signs:** `docker compose build` fails with "COPY failed: file not found".

### Pitfall 2: go.mod Module Path vs Directory Path Confusion

**What goes wrong:** After moving `daemon/` to `node/daemon/`, the go.mod still says `module github.com/gsd-build/daemon`. Developers expect `go build ./node/daemon/...` to work from root, but Go resolves modules by go.mod module path, not directory path.
**Why it happens:** Go module paths are identifiers, not filesystem paths. They don't need to match the directory structure.
**How to avoid:** Use `go.work` which maps directories to modules. From repo root, `go build ./node/daemon/...` works because go.work declares `use ./node/daemon`. The go.mod module name stays `github.com/gsd-build/daemon` unchanged.
**Warning signs:** `go: cannot find module for path` errors.

### Pitfall 3: pnpm-lock.yaml Location

**What goes wrong:** gsd-vibe has its own `pnpm-lock.yaml`. When moved to a workspace, pnpm wants a single lockfile at the workspace root.
**Why it happens:** pnpm workspaces use a single root-level lockfile. Package-level lockfiles are ignored.
**How to avoid:** Move `pnpm-lock.yaml` from `server/frontend/` to the repo root after setting up the workspace. Run `pnpm install` from root to regenerate the lockfile. Delete the old one.
**Warning signs:** `ERR_PNPM_OUTDATED_LOCKFILE` or mismatched dependency versions.

### Pitfall 4: gsd-vibe Vite Config Has Tauri-Specific Settings

**What goes wrong:** The `vite.config.ts` references `TAURI_ENV_PLATFORM`, `TAURI_ENV_DEBUG`, and sets `envPrefix: ["VITE_", "TAURI_"]`. This won't cause build failures (the env vars just won't be set), but the build target defaults to `safari15` which is fine for web.
**Why it happens:** gsd-vibe was a Tauri app.
**How to avoid:** In Phase 1, leave these as-is. The build still works because the env vars evaluate to falsy, producing reasonable defaults. Phase 4 cleans up Tauri references.
**Warning signs:** None for build -- this is cosmetic debt for Phase 4.

### Pitfall 5: Template Frontend is NOT gsd-vibe

**What goes wrong:** Someone might try to keep or merge the template's `frontend/` directory with gsd-vibe.
**Why it happens:** The template has its own React frontend (React 19, Tailwind v4, TanStack Router, biome, bun). gsd-vibe is a completely different frontend (React 18, Tailwind v3, react-router-dom, eslint, pnpm).
**How to avoid:** The template's `frontend/` directory is REPLACED entirely by gsd-vibe. Do not merge them. The template frontend's Dockerfile, nginx configs, and build scripts are discarded -- gsd-vibe brings its own.
**Warning signs:** React version conflicts, two routers, two linters.

### Pitfall 6: Template Root Uses bun, Not pnpm

**What goes wrong:** The template's root `package.json` uses `bun run --filter` commands and has a `bun.lock`.
**Why it happens:** The deployable-saas-template was designed for bun.
**How to avoid:** The template's root `package.json` is replaced with a new one using pnpm scripts. The `bun.lock` is discarded. Only `pnpm-lock.yaml` is used.
**Warning signs:** `bun` commands in scripts, stale bun.lock file.

### Pitfall 7: Git History Preservation

**What goes wrong:** Using `cp` + `rm` instead of `git mv` loses file history.
**Why it happens:** Habit, or not realizing git tracks renames.
**How to avoid:** Use `git mv` for all file moves. For directories: `git mv daemon node/daemon`. Git detects renames automatically with sufficient similarity.
**Warning signs:** `git log --follow <file>` shows no history before the move.

## Open Questions (RESOLVED)

1. **What files from deployable-saas-template root should move to server/?**
   - What we know: `docker-compose.yml`, `compose.override.yml`, `.env.example`, `pyproject.toml`, `uv.lock`, `backend/` all need to move. `compose.traefik.yml` and `copier.yml` are template-specific config.
   - What's unclear: Whether `copier.yml`, `.copier/`, `hooks/`, `.pre-commit-config.yaml`, `compose.traefik.yml` should be kept or discarded. They're template scaffolding tools, not runtime requirements.
   - Recommendation: Keep `compose.traefik.yml` (useful deployment reference). Discard `copier.yml`, `.copier/`, `hooks/` (scaffolding artifacts). Keep `.pre-commit-config.yaml` if the team uses pre-commit hooks.

2. **Should gsd-vibe's src-tauri/ directory be included?**
   - What we know: It's a Rust/Tauri backend, not used in web mode. Phase 4 removes Tauri dependencies.
   - What's unclear: Whether keeping it during Phase 1 causes build issues.
   - Recommendation: Exclude `src-tauri/` from the move entirely. It's not needed for web builds and would add confusion. The Tauri CLI devDependency (`@tauri-apps/cli`) should also be removed from package.json to avoid install failures on machines without Rust.

3. **Should the Vite version be upgraded to 8.x during this phase?**
   - What we know: STACK.md recommends Vite 8.x upgrade. gsd-vibe uses Vite 6.0.5. Success criteria says "no feature code is added or modified."
   - What's unclear: Whether a build tool upgrade counts as "feature code."
   - Recommendation: Do NOT upgrade Vite in Phase 1. The success criteria explicitly says structure and import paths only. Vite upgrade is a separate concern.
