# Phase 11.1: Cloud API Endpoints and Full Stub Wiring — Context

**Gathered:** 2026-04-13
**Status:** Ready for planning

<domain>
## Phase Boundary

Wire the frontend to real backend API endpoints for the cloud-relevant features: project listing, project detail (activity, sessions, usage tabs), and user settings persistence. Leave local-only stubs (git, file system, pickFolder, project scaffolding) as-is — they return empty state and that is acceptable.

This phase does NOT attempt to replace all 198 stubs. It targets the stubs that have real cloud equivalents or that require new backend endpoints.

</domain>

<decisions>
## Implementation Decisions

### Projects Page (D-01 to D-02)

- **D-01: Wire dashboard to GET /projects.** Replace `api.getProjectsWithStats()` call in `queries.ts`/`useProjectsWithStats` with a real call to `lib/api/projects.ts:listProjects()`. The backend returns slim data: `{id, name, node_id, cwd, user_id, created_at}`.

- **D-02: Adapt ProjectCard to slim model.** Hide or omit fields that don't exist in cloud context (`roadmap_progress`, `total_cost`, `tech_stack`, `last_activity_at`, `git_version`). Show name, node reference, working directory. Do not add backend stats computation to close this gap — clean and honest slim cards.

### Settings Persistence (D-03 to D-05)

- **D-03: Backend-persisted settings via a separate `user_settings` table.** Settings survive across devices and browsers. Chosen over localStorage (not cross-device) and JSON blob (schema enforcement preferred).

- **D-04: New Alembic migration** creates `user_settings` table with typed columns for the settings the frontend currently manages: theme, layout preferences, AI model selection, logging level, etc. Safe defaults for all existing users.

- **D-05: New FastAPI endpoints** `GET /users/me/settings` and `PUT /users/me/settings`. Wire frontend `getSettings` / `updateSettings` in `queries.ts` away from the tauri stub and to the new `lib/api/` module. Use existing `apiRequest` pattern from `lib/api/client.ts`.

### Feature Gating (D-06)

- **D-06: Leave local-only stubs as empty states.** No UI changes for git operations, pickFolder, file browser, knowledge indexing, project scaffolding, GitHub API, scanner summary, PTY create/attach. These stubs already return `[]`, `null`, or `false`. The UI shows empty states. No hiding, no disabling, no tooltips. This is acceptable for cloud context.

### Project Detail Page Tabs (D-07 to D-10)

- **D-07: Keep all tabs visible.** Wire the tabs that have real backend equivalents; leave the rest showing empty state from their stubs. No tabs are hidden or removed.

- **D-08: Activity feed tab** — Wire `activity-feed.tsx` to `GET /activity?project_id=X`. The backend endpoint exists in `activity.py` (`GET /activity` with optional `project_id` filter). Wire `onActivityLogged` to the SSE stream at `GET /activity/stream`.

- **D-09: Sessions tab** — Wire to `GET /sessions?project_id=X` (`sessions.py` exists). Show list of past/active sessions on this project: session id, node, status, start time.

- **D-10: Usage / Cost tab** — Wire to `GET /usage?project_id=X` (`usage.py` exists). Show per-session token costs for this project.

### Claude's Discretion

- Exact column names and schema for `user_settings` table — planner decides based on current `Settings` type in `lib/tauri.ts`
- Whether to add `project_id` filter to existing `/usage` and `/sessions` endpoints or create new query params — follow existing patterns in those route files
- Frontend query key naming for new settings queries — follow `queryKeys` patterns in `lib/query-keys.ts`

</decisions>
