# Phase 11.1: Cloud API Endpoints and Full Stub Wiring - Research

**Researched:** 2026-04-13
**Domain:** FastAPI backend endpoints + React frontend API wiring
**Confidence:** HIGH

## Summary

This phase wires the frontend to real backend API endpoints for cloud-relevant features: projects listing (slim model), user settings persistence, activity feed, sessions-by-project, and usage-by-project. The backend already has working endpoints for activity, sessions, and usage, but they lack `project_id` filtering. A new `user_settings` table and endpoints need to be created from scratch.

The most significant architectural finding is that `SessionModel` has no `project_id` foreign key -- sessions are linked to nodes/users but not projects. Since D-08/D-09/D-10 require filtering by project, a `project_id` column must be added to `SessionModel` (and propagated to the session creation flow), OR queries must join through `Session.node_id == Project.node_id AND Session.cwd == Project.cwd`. Adding a FK is cleaner and more performant.

**Primary recommendation:** Add `project_id` FK to `SessionModel` in the same Alembic migration that creates `user_settings`. Wire all frontend queries through existing `lib/api/` modules using the established `apiRequest` pattern.

## Architecture Patterns

### Critical Finding: SessionModel Has No project_id

`SessionModel` (models.py:226) has `user_id` and `node_id` but NO `project_id`. The `Project` model (models.py:275) has `node_id` and `cwd`. Sessions also have `cwd`. [VERIFIED: server/backend/app/models.py]

**Two approaches to filter sessions/activity/usage by project:**

1. **Add `project_id` FK to SessionModel (RECOMMENDED):** Clean, indexable, explicit. Requires migration + updating `crud.create_session()` to accept optional `project_id`. Sessions created before the migration get `project_id=NULL` -- acceptable since they predate the project concept.

2. **Join on node_id + cwd:** No migration needed, but fragile (cwd could differ by trailing slash, case, etc.), slower queries, and semantically wrong -- a session on node X at path Y isn't necessarily part of a project.

**Recommendation:** Add `project_id: uuid.UUID | None = Field(default=None, foreign_key="project.id", index=True)` to `SessionModel`. This is Claude's discretion per CONTEXT.md. [ASSUMED]

## Common Pitfalls

### Pitfall 1: ProjectWithStats vs ProjectPublic Type Mismatch
**What goes wrong:** `ProjectCard` component expects `ProjectWithStats` (16+ fields). `ProjectPublic` from the API has 6 fields. TypeScript will catch this at compile time but the runtime rendering will show `undefined` for missing fields.
**Why it happens:** The dashboard page was built for a Tauri desktop app with rich local data. Cloud projects have slim server-side data.
**How to avoid:** Create a slim cloud variant of ProjectCard, or map `ProjectPublic` to a `ProjectWithStats`-like object with explicit nulls/defaults for missing fields. D-02 says "show name, node reference, working directory" -- don't try to fill the rich card template.
**Warning signs:** TypeScript errors on `project.roadmap_progress`, `project.total_cost`, `project.path`.

### Pitfall 2: No project_id on SessionModel
**What goes wrong:** D-09 and D-10 ask for sessions/usage filtered by `project_id`, but SessionModel has no such column. A naive approach would be to join on `node_id + cwd`, but this is fragile.
**Why it happens:** Sessions were designed around nodes, not projects. Projects were added later.
**How to avoid:** Add `project_id` FK migration. Backfill NULL for existing sessions. Update `create_session` to accept optional `project_id`.
**Warning signs:** Empty results when filtering by project, because no sessions have project_id set.

### Pitfall 3: Two Activity Feed Implementations
**What goes wrong:** The codebase has TWO activity feed implementations: (1) `use-activity-feed.ts` (global sidebar, already cloud-wired), and (2) `activity-feed.tsx` component using `useActivityLog` from `queries.ts` (project-level, still using stub). Wiring only one leaves the other broken.
**Why it happens:** The global activity sidebar was wired during Phase 5. The project-level activity feed component was not.
**How to avoid:** Wire the project-level `useActivityLog` in `queries.ts:395-400` to call the real API with `project_id` filter. Leave the global `use-activity-feed.ts` as-is.
**Warning signs:** Global activity works but project detail activity tab shows nothing.

### Pitfall 4: Settings Endpoint Returns null for New Users
**What goes wrong:** `GET /users/me/settings` returns 404 for users who haven't saved settings yet, causing React Query to error and the settings page to show an error state.
**Why it happens:** New users have no row in `user_settings`.
**How to avoid:** `GET /users/me/settings` should create-on-read: if no row exists, insert one with defaults and return it. Alternatively, return a default settings object without persisting.
**Warning signs:** Settings page errors on first login.

### Backend: Settings Endpoints (users.py)
```python
# Source: follows existing users.py patterns (CurrentUser dep, SessionDep)
@router.get("/me/settings")
def get_user_settings(session: SessionDep, current_user: CurrentUser) -> dict:
    settings = session.exec(
        select(UserSettings).where(UserSettings.user_id == current_user.id)
    ).first()
    if not settings:
        # Create-on-read with defaults
        settings = UserSettings(user_id=current_user.id)
        session.add(settings)
        session.commit()
        session.refresh(settings)
    return settings.model_dump(exclude={"id", "user_id"})

@router.put("/me/settings")
def update_user_settings(
    session: SessionDep, current_user: CurrentUser, body: UserSettingsUpdate
) -> dict:
    settings = session.exec(
        select(UserSettings).where(UserSettings.user_id == current_user.id)
    ).first()
    if not settings:
        settings = UserSettings(user_id=current_user.id)
    update_data = body.model_dump(exclude_unset=True)
    settings.sqlmodel_update(update_data)
    settings.updated_at = get_datetime_utc()
    session.add(settings)
    session.commit()
    session.refresh(settings)
    return settings.model_dump(exclude={"id", "user_id"})
```

## Open Questions

1. **SessionModel project_id backfill**
   - What we know: Existing sessions have no project_id. New sessions created via POST /sessions don't pass project_id.
   - What's unclear: Should existing sessions be backfilled by matching node_id+cwd to projects? Or just leave them NULL?
   - Recommendation: Leave NULL. Only new sessions created after this phase get project_id. The frontend should handle NULL gracefully.

2. **Session creation flow change**
   - What we know: POST /sessions accepts `{node_id, cwd}`. Projects are identified by `{node_id, cwd}` pair.
   - What's unclear: Should the frontend pass `project_id` when creating sessions, or should the backend infer it from node_id+cwd?
   - Recommendation: Backend infers project_id from node_id+cwd at session creation time. No API change needed for session creation.
