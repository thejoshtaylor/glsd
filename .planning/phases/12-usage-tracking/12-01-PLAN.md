<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Node WebSocket -> Server | Cost data from daemon arrives via authenticated node connection; values could be manipulated by a compromised node |
| Browser -> REST API | Usage endpoints accessed via JWT cookie auth; must filter by user_id |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-12-01 | Information Disclosure | GET /usage/, GET /usage/summary | mitigate | All queries filter by `current_user.id` from JWT -- never accept user_id from query params. Test: test_usage_isolation verifies user A cannot see user B data. |
| T-12-02 | Tampering | period query param on usage endpoints | mitigate | FastAPI Query validation with regex `^(7d|30d|90d|all)$` rejects arbitrary input. |
| T-12-03 | Tampering | cost_usd from node WebSocket | accept | Cost data comes from authenticated node connection; node ownership verified at session creation. A rogue node could send false cost data for its own sessions, but this is inherent to the trust model (node owner controls node). No additional mitigation needed. |
| T-12-04 | Denial of Service | UsageRecord insert failure | mitigate | Separate DB session for usage write; failure does not crash WebSocket handler or roll back session status update. Wrapped in try/except with logging. |
| T-12-05 | Spoofing | Unauthenticated access to /usage endpoints | mitigate | Both endpoints require `CurrentUser` dependency (JWT cookie auth). Test: unauthenticated request returns 401. |
</threat_model>
